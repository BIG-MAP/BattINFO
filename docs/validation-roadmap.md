# Validation Roadmap

BattINFO needs validation strong enough that downstream systems can treat the package as a contract-bearing platform dependency, not just a collection of JSON examples and helper scripts.

This roadmap defines the implementation sequence for that work.

## Objectives

- Make validation deterministic across CLI, API, saving, publishing, and indexing.
- Guarantee machine-readable failure output with stable issue codes.
- Enforce structural, referential, semantic, and publication-layer correctness.
- Support strict CI gates while still allowing staged cleanup of existing data.

## Current Baseline

Current validation already provides:

- JSON Schema validation for multiple profiles.
- Some save-time cross-reference checks.
- Limited JSON-LD term validation.
- CI smoke validation for canonical examples and generated datasheets.

Current gaps include:

- Schema `format` keywords are not a guaranteed runtime gate unless explicitly enabled.
- Validation logic is split across modules and workflows.
- JSON-LD validation is shallow.
- Domain semantics are not centrally enforced.
- Error reporting is mostly string-based and inconsistent.

## Design Principles

- One validation engine, many policies.
- Structured results first, human-readable rendering second.
- Fail closed for hard contract violations.
- Distinguish schema, reference, semantic, and publication issues.
- Keep policy versioned so downstream systems can pin behavior.

## Validation Layers

### 1. Schema Validation

Purpose:
- Validate shape, required fields, enums, patterns, bounds, and formats.

Scope:
- All canonical records.
- Authoring profiles such as `battery-descriptor`, `cell-type-datasheet`, and ingest manifests.

Required outcomes:
- Enforce JSON Schema formats such as `uri`, `email`, `date`, and `date-time`.
- Use one shared validator factory across package entry points.
- Preserve full error lists for batch/report modes.

### 2. Reference Validation

Purpose:
- Validate that IDs point to existing records of the correct type.

Scope:
- Cell-instance to cell-type.
- Test to cell-instance and dataset.
- Dataset to related cell/test resources.
- Library/profile references where applicable.

Required outcomes:
- Support validation against a source tree, built index, or frozen registry snapshot.
- Emit typed issues like `missing_reference` and `wrong_reference_type`.

### 3. Semantic Validation

Purpose:
- Validate domain meaning, not just document shape.

Scope:
- Controlled values.
- Unit and property compatibility.
- Numeric sanity ranges.
- Min/max ordering.
- Provenance completeness.
- Identifier and version consistency.

Required outcomes:
- Rules are grouped by resource type and strictness policy.
- Warnings can be promoted to errors as data quality improves.

### 4. Publication Validation

Purpose:
- Ensure published artifacts are safe for downstream graph and resolver consumers.

Scope:
- JSON-LD generated by mapping and publication flows.
- Resolver artifact completeness.

Required outcomes:
- Parse/expand JSON-LD successfully.
- Validate expected identifiers and type assignments.
- Add graph-shape validation for publication payloads.

## Implementation Phases

### Phase 1: Foundation

Deliverables:
- Shared validator factory with enforced schema formats.
- Roadmap and validation contract documentation.
- Negative tests for format failures.
- Consistent schema validation behavior in `validate_json()` and API write paths.

Acceptance criteria:
- Invalid URLs and timestamps fail in both profile validation and save flows.
- No public write path bypasses schema validation when `validate=True`.

### Phase 2: Unified Validation Engine

Deliverables:
- `src/battinfo/validate/core.py`
- Structured result types:
  - `ValidationIssue`
  - `ValidationReport`
  - `ValidationPolicy`
- Shared entry points:
  - `validate_record`
  - `validate_file`
  - `validate_batch`

Acceptance criteria:
- CLI, API, save, publish, and index paths use the same engine.
- Machine-readable validation output is available for automation.

### Phase 3: Referential Integrity

Deliverables:
- `src/battinfo/validate/references.py`
- Resource-graph checker against repository state or index snapshot.
- Stable issue codes for missing and mismatched references.

Acceptance criteria:
- Broken references are caught outside save flows as well.
- Batch validation reports all missing references in one run.

### Phase 4: Semantic Rules

Deliverables:
- `src/battinfo/validate/semantic.py`
- Rule sets for:
  - property/unit compatibility
  - quantity consistency
  - controlled vocab alignment
  - identifier normalization
  - provenance completeness

Acceptance criteria:
- Semantic contradictions are surfaced with stable codes and paths.
- Rules can run in `warn` or `error` mode.

### Phase 5: Publication and Graph Validation

Deliverables:
- `src/battinfo/validate/publication.py`
- JSON-LD parse validation.
- Publication artifact validation.
- Optional SHACL or equivalent graph-shape checks.

Acceptance criteria:
- Generated publication artifacts are validated as part of CI/release flows.
- JSON-LD validation is deeper than allowed-term checking.

### Phase 6: Downstream Contract and Compatibility

Deliverables:
- Public issue catalog and stability policy.
- Validation policies such as `core`, `strict`, `publisher`, `ingest`.
- Documentation for embedders and CI consumers.

Acceptance criteria:
- Downstream systems can pin a validation policy and rely on stable behavior.

## Proposed Module Layout

- `src/battinfo/validate/core.py`
- `src/battinfo/validate/schema.py`
- `src/battinfo/validate/references.py`
- `src/battinfo/validate/semantic.py`
- `src/battinfo/validate/publication.py`
- `src/battinfo/validate/render.py`

Existing modules should migrate toward this structure rather than continue to accrete ad hoc checks.

## Issue Model

Recommended minimum fields:

- `code`
- `severity`
- `path`
- `message`
- `hint`
- `validator`
- `resource_type`
- `profile`

Example issue codes:

- `schema.required`
- `schema.format.uri`
- `schema.format.date_time`
- `reference.missing`
- `reference.type_mismatch`
- `semantic.unit_mismatch`
- `semantic.range_invalid`
- `semantic.version_conflict`
- `publication.jsonld_invalid`
- `publication.shape_invalid`

## CI Rollout Strategy

Stage 1:
- Enforce schema-format failures and preserve existing gates.

Stage 2:
- Add repo-wide validation job with structured report artifact.

Stage 3:
- Add referential integrity as required CI for canonical resources.

Stage 4:
- Add semantic and publication validation to release/publish workflows.

## Immediate Backlog

1. Enforce schema `format` validation everywhere schema validation is used.
2. Create a shared validator factory to remove duplicated validator setup.
3. Add negative tests for malformed `uri`, `email`, `date`, and `date-time`.
4. Introduce structured validation issue types without breaking existing CLI/API ergonomics.
5. Refactor current save reference checks into reusable validation functions.
6. Define the first semantic rule pack for quantities and controlled values.

## Definition of Done

BattINFO validation is robust enough for downstream systems when:

- all public write and publish paths use one validation engine,
- structured issue reports are stable and documented,
- schema formats and references are enforced consistently,
- semantic and publication validation exist for core resources,
- CI exercises both positive and negative fixtures,
- downstream clients can pin a policy and trust pass/fail behavior.
